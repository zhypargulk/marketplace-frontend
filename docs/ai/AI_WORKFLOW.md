# AI Workflow (Marketplace Frontend)

Кратко: какими AI-инструментами пользовались, как строили итерации, какие проверки выполняли, ключевые инженерные решения и их причины (§8.1).

---

## 1. Какими AI-инструментами пользовались

- **Cursor** — генерация компонентов Vue, роутинга, слоёв FSD, типов, стилей.
- **Роль инженера:** формулирует задачу и контракты с API, проверяет результат (ручные сценарии, ревью кода), вносит правки и доводит до production-ориентированного уровня.

---

## 2. Как строили итерации: план → генерация → проверка → исправление

| Этап | План | Генерация (AI) | Проверка | Исправление |
|------|------|----------------|----------|-------------|
| **Структура** | Vue 3 + TypeScript, FSD: app, pages, widgets, features, entities, shared. | Папки, router, App.vue, заглушки страниц. | Соответствие слоям, импорты. | Перенос кода между слоями при необходимости. |
| **Публичные страницы** | Список товаров (пагинация/бесконечный скролл), детали (атрибуты, офферы, seller name/rating, сортировка). | ProductsListPage, ProductDetailsPage, запросы к API. | Ручной прогон: список, детали, сортировка. | Правка типов, плейсхолдер при отсутствии фото. |
| **Admin** | Логин, список продуктов (Add/Edit/Delete), форма продукта (атрибуты, изображение), офферы (add/delete, upload image). | AdminLoginPage, AdminProductsPage, AdminProductFormPage, AdminOffersPage. | Логин, CRUD, загрузка изображения вручную. | Исправление маршрутов и форм. |
| **Auth и shared** | Единый источник правды для токена; shared config и API client. | useAuth (token, setToken, authHeaders), shared/config, shared/api. | Обновление UI после логина без перезагрузки. | Синхронизация использования useAuth во всех страницах. |
| **UX** | При отсутствии фото — плейсхолдер «Product photo»; не дергать image endpoint, если API вернул null. | Условия отображения img/placeholder, hasImage из thumbnail_url в списке. | Проверка списка и детальной страницы без изображений. | Точечные правки разметки и условий. |

Каждая итерация — отдельный запрос с чёткой задачей; результат проверяется в браузере и по коду перед следующим шагом.

---

## 3. Какие проверки выполняли

- **Тесты:** визуальная и ручная проверка (при необходимости можно добавить Vitest + Vue Test Utils для критичных компонентов).
- **Линтеры:** ESLint по проекту; проверка типов TypeScript.
- **Ручные сценарии:** `npm run dev`, проход по сценариям: каталог → детали товара → сортировка офферов; логин в Admin → список продуктов → создание/редактирование продукта → офферы → загрузка изображения; проверка плейсхолдера «Product photo» при отсутствии изображения.

Исправления задаются точечными промптами (например: «добавить seller_name и seller_rating в карточку оффера», «показывать плейсхолдер, если thumbnail_url null»).

---

## 4. Ключевые инженерные решения и их причины

| Решение | Причина |
|---------|---------|
| **Feature-Sliced Design (app, pages, widgets, features, entities, shared)** | Масштабируемость, понятные границы слоёв, соответствие ТЗ; упрощает онбординг и дальнейшее расширение (фильтры, дашборды). |
| **useAuth с общим состоянием токена (setToken при логине)** | Header и защищённые страницы реагируют на логин без перезагрузки; один источник правды для Authorization. |
| **shared/config и shared/api** | Единая точка для base URL и axios; проще менять окружение и мокать в тестах. |
| **Плейсхолдер «Product photo» при отсутствии изображения** | Понятный UX; не запрашивать image endpoint, если API вернул image_url/thumbnail_url null — меньше запросов и корректное отображение. |
| **Маршруты admin: /new и /:id/edit с одной формой** | Меньше дублирования кода, единая логика валидации и отправки. |

Материалы позволяют убедиться, что фронт создавался и эволюционировал управляемо, с контролем качества (ревью, ручные сценарии) — критерий приёмки §8.2.
